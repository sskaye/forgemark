Forgemark · Design rationale

The file is the document. The UI is a lens on it.

Two pages on the calls we made designing Forgemark, what we changed our minds about, and what the brief didn’t account for.

v1 · macOS · prepared for the proposal authors

The throughline

The brief gave us five pillars; one of them ended up doing more work than the others. The file is the source of truth stops being a design pillar and starts being a constraint the moment you sit with the format. Every choice downstream — the lost-anchor recovery flow, the source view, how AI authors are presented, even the Clean Export confirmation copy — comes from taking that pillar seriously.

Practically: the markdown file is plain text that humans, AI agents, and external editors will all touch. The UI is a lens on a file that may have changed underneath it. That reframes most of the screens.

The UI doesn’t own the data. It renders the file, helps you change it, and accepts that other tools will change it too.

What we held tight on

Native, quietly

SF Pro everywhere, system mono for source, traffic-light chrome, system-blue accent that respects the user’s appearance preference. Tauri was chosen specifically to feel native and we did not want to spend that affordance on brand expression. The wordmark is a single bracketed pilcrow — present in the title bar and About dialog and almost nowhere else.

AI authors are peers, not patients

Same avatar treatment. No badge. No glyph. Identity is a name. We considered a small monogram or model glyph in the avatar and rejected it: the moment you put a marker on AI comments, you’re telling the user to read those differently, and the brief is explicit that we are not doing that. If a reviewer needs to skim only human comments, the existing author-filter handles it without the design taking a position.

Anchor highlights are quiet, but distinct

The reviewer asked for distinct over subtle, and we agreed once we saw both side-by-side in dark mode. Subtle reads as decorative; distinct reads as functional. We held the chroma low so a paragraph with five anchors still reads as prose, not as a chart.

What we changed our minds about

The orphan card

Initial draft called drift-detached comments orphaned, with a purple "?" icon. Two problems: orphaned is jargon you have to learn, and the icon read as “an unanswered question someone asked,” not “the system can’t locate the passage this is about.” Renamed to Lost anchor. The label is a noun phrase that says exactly what is wrong, in two short words, with no verb. The icon stays — but the copy carries the load.

Resolved comments

We went back and forth on hide-by-default vs. collapse-in-place. Collapse-in-place won. Hidden-by-default makes the sidebar lighter but it punishes the most common second action after resolving (reopening one minute later, often because a reply came in or because you were wrong). A one-line collapsed card is the right cost.

Tweaks vs. product

The first prototype exposed seven Tweaks: theme, pairing, highlight intensity, density, font size, show-resolved, identity. Five of those were design decisions disguised as preferences. They’re now locked. Theme, font size, and author identity remain tweakable because users genuinely have a stake in them; the rest are the design.

The five open questions, answered

  1. Q1 Right-pinned, not detachable. A second window doubles the cognitive surface for marginal benefit. Reviewers think paragraph-by-paragraph; the sidebar belongs next to the paragraph.
  2. Q2 Resolved comments collapsed in place, not hidden. Reopen-rate is high enough that hiding penalizes the recovery action. Collapsed one-line cards keep the sidebar quiet without losing the breadcrumb.
  3. Q3 Per-document, default rendered. A global preference loses you the file-by-file context (a code-heavy file might warrant source view by default). Per-document state, no global override, default to rendered.
  4. Q4 Skipped for v1. Body diff is a real concern but it’s a different product surface. Stub note in the rationale; revisit when AI agents are doing this often enough to pattern-match.
  5. Q5 Dashed magenta underline; banner above the prose; sidebar section pinned to top. Yellow is anchor, green is suggestion, red would alarm, blue is focus. Magenta is the only unused slot in the palette and reads as “needs your eye” without claiming severity. Dashed (vs. solid) reinforces uncertainty. The banner gives a recovery action without forcing a modal on file-open.

Pushback

Issue 1 · the source view is not symmetric with rendered view

Pillar #2 says raw-source should feel like a peer view, not an admin escape hatch. We held to that visually — same layout, same scroll position, anchored cards stay clickable from source — but in practice some interactions only make sense in rendered view. You can’t select a passage to comment in source mode (the YAML markers would land in the wrong place); accept-suggestion in source would need to rewrite both the prose AND the YAML block in one operation. Source view is currently read-mostly. We think this is correct, but we did not call it out as a divergence from the pillar. Flagging.

Issue 2 · the format does not survive a rename

Comments anchor by inline marker pair AND by anchor_text snapshot AND by context windows. None of these survive a passage being deleted entirely. The lost-anchor flow handles drift, not deletion. If a reviewer deletes a paragraph that had three comments, those three comments become lost-anchor cards with no surviving prose to match against. The current Reattach modal would offer 0% matches. Either (a) lost-anchor cards need a "no candidates — keep as note?" path, or (b) deletions need a confirmation when the passage carries comments. We’ve mocked (a) implicitly via the Discard button, but (b) is not in v1.

Issue 3 · the brief assumes one document

Every screen we’ve drawn is single-document. The proposal does not address how reviewers move between files in a session. There is no recents list, no file switcher, no project concept. Macros open many files in a workday and Forgemark’s current chrome punts on this entirely. Decision needed: do we ship single-file v1 with reviewers re-using ⌘O, or do we add a sidebar drawer with recent files? We chose the former and want explicit alignment.

Issue 4 · simultaneous human + AI editing is undefined

If an AI agent edits the file while a human has it open in Forgemark, what happens? The proposal’s anchor-drift recovery handles open-after-edit; it does not handle edit-during-open. We’ve assumed Tauri-side file watching detects the change and offers a reload prompt, but the UX of "your unsaved comment composer will be lost" is not designed. This intersects with the v1 multi-author exclusion in the brief. Calling it out so it doesn’t surface as a v1.1 surprise.

What’s in the box

  • Forgemark.html — interactive prototype. All six flows are wired (select-to-comment, reply, edit, suggest-edit accept/reject, resolve, reattach). Light/dark theme, prose-size slider, author-identity switcher exposed as Tweaks.
  • Forgemark — Design System.html — tokens, type pairings, highlight matrix in light + dark, the full card-state matrix, plus First Run, Settings, Empty State, Clean Export, and the right-click Comment menu. Every state in the brief is enumerated and labelled.
  • Forgemark — Storyboard.html — the six flows as freeze-frames with beats.
  • This document.

What’s explicitly not in the box

  • The application icon (.icns / .ico) — wordmark candidates exist; production export is a build-pipeline task.
  • Full keyboard-shortcut reference card — shortcuts are present in the menu bar mock; a separate sheet would be useful for the help screen.
  • Settings flows for the .skill download — affordance exists in the Settings mock; the link target depends on engineering.
  • Body-edit diff view — see open question 4. Skipped per direction.
